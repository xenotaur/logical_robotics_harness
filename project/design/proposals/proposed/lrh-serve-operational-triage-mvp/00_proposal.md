# LRH Serve Operational Triage MVP

## Summary

**LRH Serve Operational Triage MVP** is a safe-default,
read-only/meta-aware dashboard and prompt workbench for deciding and
launching human-gated prompt-driven work.

The MVP is an operational triage surface for LRH's existing project
control plane. It helps a human open `lrh serve`, understand which
registered projects need attention, inspect project-local focus and
planning state, identify ready or readiness-deficient leaves, and obtain
the same human-gated prompts that LRH request/rendering commands would
produce. The MVP intentionally remains a local-assist viewer and prompt
preparation surface rather than an autonomous runner or repository
mutation engine.

Target workflow:

```text
Open lrh serve
→ see registered projects in operational swimlanes
→ choose a project needing attention
→ inspect focus, validation, designs, workstreams, and work items
→ identify execution-ready or readiness-deficient leaves
→ preview/copy/download the correct human-gated prompt or repair prompt
→ execute elsewhere
→ later record evidence/execution/run state through explicit follow-up mechanisms
```

This proposal captures the implementation architecture and sequencing for
future focused PRs. It does not implement the UI, add routes, change CLI
behavior, or create write-capable serve endpoints.

## Problem / motivation

LRH's control plane increasingly models principles, focus, designs,
workstreams, work items, execution readiness, run packets, and evidence.
Those artifacts are human-readable, but a maintainer still has to inspect
multiple files and commands to answer operational questions:

- Which registered project should be handled next?
- Is the project valid, stale, unavailable, blocked, or ready for a
  prompted manual action?
- Which design or workstream explains the next leaf?
- Is a work item execution-ready, or is it missing readiness fields?
- Which prompt renderer should be used, and what source inputs did it
  consume?
- Is a missing feature an actual error, a planned capability gap, an
  unsafe/deferred affordance, or an unknown state?

A safe triage surface should reduce that lookup cost without changing
the authority model. Project-local `project/` directories must remain the
source of truth. Meta workspace state can coordinate and summarize, but
it must not override project-local precedence or invent independent
planning semantics.

## Goals

- Provide a deterministic operational dashboard for registered LRH
  projects using evidence-backed swimlanes.
- Project shared control-plane, planning, meta, and assist data into
  serve view models without duplicating workflow logic in routes or
  templates.
- Show validation state, current focus, design/workstream/work-item
  traceability, readiness, capability gaps, and source links.
- Provide a prompt workbench that previews, copies, and downloads the
  same human-gated prompt text generated by the appropriate `lrh
  request` renderer.
- Provide repair/readiness prompt affordances when work items are not
  ready.
- Make missing or future features explicit first-class capability gaps
  instead of broken UI.
- Preserve safe defaults: local binding, read-only behavior, no external
  calls, no arbitrary filesystem browsing, and no repository mutation.
- Keep stages small enough that each implementation PR can be reviewed
  and validated independently.

## Non-goals

- Do not dispatch agents, model-provider calls, local coding agents, or
  remote workflow jobs from `lrh serve` in this MVP.
- Do not create branches, commits, pull requests, merges, releases, or
  publications.
- Do not add autonomous execution, stabilization loops, observation
  adapters, backend adapters, or mutation-capable run behavior.
- Do not replace `lrh validate`, `lrh snapshot`, `lrh request`, or
  project-local Markdown documents as the controlling interfaces.
- Do not make the meta workspace authoritative for project-local
  precedence.
- Do not save edited prompts, update execution records, mutate work
  items, or write run state unless a later explicit write-boundary design
  is adopted with tests.
- Do not over-specify pixel-perfect UI details in this proposal; visual
  treatment should remain compatible with the LRH Console visual-language
  proposal.

## Design principles

1. **Project-local authority first.** A project's `project/` directory is
   authoritative for that project's planning and status. Meta state is
   coordinating and informative.
2. **Shared semantics, single implementation.** `lrh serve` must render
   shared view models produced by control-plane, planning, meta, and
   assist APIs. It must not reinterpret readiness, precedence, lane
   membership, or prompt inputs independently.
3. **Read-only by default.** The MVP is a local-assist surface that
   prepares human decisions and artifacts. It does not mutate
   repositories.
4. **Evidence-backed state.** Operational lanes and status badges should
   cite validation, project-control artifacts, readiness data, execution
   records, run state, or explicit capability gaps.
5. **Capability gaps are product state.** Planned, unavailable, blocked,
   unsafe/deferred, and unknown affordances should be visible and
   actionable without being styled as catastrophic failures.
6. **Human-gated prompt work.** Prompt generation helps humans act in a
   separate execution environment. Launch and follow-up recording remain
   explicit human actions.
7. **Accessible redundant cues.** Status must be visible through text and
   structure, not color alone.

## High-level architecture

The MVP should use a layered design. Current package names may evolve,
but the conceptual boundary is:

```text
src/lrh/control_plane/
  loaders, validators, typed artifacts

src/lrh/planning/ or current equivalent
  planning relationship/index model
  workstream/work-item/design relationships
  active leaves and readiness summaries

src/lrh/meta/
  workspace context
  project registry
  project locator resolution

src/lrh/assist/
  prompt rendering
  request templates
  run packet/report renderers where available

src/lrh/serve/
  HTTP server
  routes
  HTML rendering
  static assets
  view-model projection only
```

The `serve` layer should be thin. Routes should resolve a request, call
shared APIs, render escaped HTML, and expose static assets. They should
not duplicate loader, validation, planning, readiness, request-rendering,
or capability-gap logic.

The view-model projection can live at the seam between shared LRH APIs
and `src/lrh/serve/`, but the source of each field should remain clear:
control-plane validation, planning relationships, meta registry state,
assist prompt rendering, or explicit capability-gap projection.

## Information architecture

The MVP should expose these route/page concepts. Exact route paths can be
chosen during implementation, but they should remain deterministic and
linkable.

### Meta dashboard / swimlanes

The meta dashboard groups registered projects into operational swimlanes.
Initial deterministic lane concepts:

- **Needs Attention** — validation errors, safety issues, missing
  authority, or other human-attention state is present.
- **Needs Validation** — state is available but has not been validated,
  validation is stale, or the source changed since the last known
  validation.
- **Ready for Prompted Work** — one or more leaves have sufficient
  readiness and prompt inputs for manual prompted work.
- **Active / In Progress** — project state indicates active execution,
  in-progress work items, or open manual run state.
- **Paused / Blocked** — explicit blockers, pauses, or missing upstream
  decisions prevent safe progression.
- **Stable / Complete** — no active actionable leaves are known, and
  validation/source state is healthy enough to display.
- **Unknown / Unavailable** — project source, registry entry, or required
  state cannot be resolved.

Lane names can evolve, but the classifier must be deterministic,
documented, evidence-backed, and based on shared state. The page should
show lane labels, counts, and short explanations so a human understands
why a project appears in a lane.

### Project dashboard

A project dashboard drills into one registered project. It should show
validation summary, current focus, design summary, workstream summary,
work item summary, ready and not-ready leaves, capability gaps, and
source/detail links.

Project cards on the meta dashboard should show, when available:

- project display name / short name;
- locator path or URL;
- source state: live, cached, or unavailable;
- validation status;
- current focus;
- active workstream count;
- active work item count;
- execution-ready or manual-prompt-ready leaf count;
- readiness-deficient leaf count;
- adopted-but-not-implemented design count;
- capability gaps; and
- source/detail links.

### Design detail view

The design detail view should show proposal lifecycle state,
implementation status, related workstreams/work items/evidence, adoption
or supersession metadata, source links, and related capability gaps. It
should rely on the lifecycle metadata model rather than inspecting path
names alone.

### Workstream detail view

The workstream detail view should show workstream lifecycle/stage, linked
designs, child work items, readiness summary, active leaves, blockers,
evidence expectations, source links, and capability gaps.

### Work item detail view

The work item detail view should show type, status, readiness, missing
readiness fields, related design/workstream/focus context, source links,
and prompt affordances. Ready work items should link to prompt preview,
copy, and download actions. Not-ready work items should show the missing
fields and offer a repair/readiness prompt affordance.

### Prompt workbench

The prompt workbench is a transient human-assist page for one promptable
leaf or repair task. It should show the generated prompt, equivalent CLI
command, source inputs, missing inputs, and copy/download controls.

## Safe-default behavior and security/privacy boundaries

For this MVP, `lrh serve` is a read-only/local-assist control surface:

- It is not an autonomous runner.
- It does not dispatch agents.
- It does not create branches, commits, pull requests, merges, releases,
  or publications.
- It does not mutate repositories by default.
- It must not reinterpret planning semantics independently of shared LRH
  APIs.
- Project-local `project/` directories remain authoritative.
- Meta workspace state is coordinating/informative, not an override of
  project-local precedence.

Security and privacy constraints:

- Bind locally by default, for example `127.0.0.1`, consistent with
  existing `lrh serve` behavior.
- Do not claim localhost is a security sandbox.
- Do not expose secrets, tokens, environment variables, private runtime
  state, or unrequested file contents.
- Do not browse arbitrary filesystem paths. Project lookup must come from
  explicit serve input, project-root selection, or registered meta
  project locators.
- Do not make external network calls from the server for MVP behavior.
- Do not add write endpoints for this design's MVP.
- If future explicit-click writes are added, they need a separate design
  boundary, threat model, UI language, and tests.

## View-model / contract sketch

The exact Python API can be designed in the implementation PR, but the
serve surface should consume stable, serializable view models with clear
provenance. A conceptual shape:

```text
ServeWorkspaceView
  generated_at
  workspace_mode
  projects: list[ProjectOperationalCard]
  lanes: list[OperationalLane]
  capability_gaps: list[CapabilityGap]

OperationalLane
  id
  label
  explanation
  project_count
  project_ids
  evidence_refs

ProjectOperationalCard
  project_id
  display_name
  short_name
  locator
  source_state
  validation_summary
  current_focus_summary
  active_workstream_count
  active_work_item_count
  ready_leaf_count
  readiness_deficient_leaf_count
  adopted_not_implemented_design_count
  capability_gaps
  source_links
  detail_url

ProjectOperationalDetail
  project_card
  validation
  focus
  designs
  workstreams
  work_items
  ready_leaves
  not_ready_leaves
  prompt_affordances
  capability_gaps

PromptWorkbenchView
  target_kind
  target_id
  renderer_id
  prompt_text
  equivalent_cli
  source_inputs
  missing_inputs
  allowed_actions
  disabled_actions
  safety_notice
```

Contracts should avoid raw dictionaries as the long-term internal API
when typed models are appropriate. HTML templates can receive simple
serialized projections, but the authoritative interpretation should live
in shared typed objects.

## Capability gaps / not-implemented affordances

Missing or planned features should be modeled as first-class capability
gaps, not as broken UI. The projection should distinguish at least:

- `unavailable` — required source cannot be reached or resolved;
- `not_implemented` — design exists but the runtime affordance is not
  built yet;
- `blocked` — an explicit blocker prevents safe progression;
- `unsafe_deferred` — an affordance would be unsafe without a later
  explicit boundary; and
- `unknown` — LRH cannot classify the state with available evidence.

Each capability gap should include a human label, explanation, severity
or presentation class, source evidence, and links to the most relevant
design document, workstream, work item, execution record, or follow-up
prompt path when available.

Red/error styling should be reserved for actual validation errors,
safety issues, broken state, or failed source resolution. Planned-but-not-
yet-implemented features should remain visible but should not be
presented as catastrophic failures.

## Prompt workbench behavior

For ready work items, the workbench should preview, copy, and download
the same prompt text generated by the appropriate `lrh request` renderer.
The workbench must not maintain a second prompt dialect. It should show
which renderer and source inputs produced the prompt.

For not-ready work items, the workbench should show missing readiness
fields and provide a repair/readiness prompt affordance. The repair
prompt is still human-gated and should make missing inputs explicit.

The prompt workbench may:

- preview generated prompt text;
- allow transient browser editing;
- copy prompt text to the clipboard;
- download Markdown;
- show the equivalent CLI command;
- show source inputs used; and
- show missing inputs.

The prompt workbench must not:

- send prompts to an agent;
- create branches;
- create commits;
- create pull requests;
- update execution records automatically;
- mutate work items automatically; or
- save prompts unless a later explicit write boundary is designed and
  approved.

Transient browser edits should be clearly labeled as local unsaved text.
If a future design adds save/export/write behavior beyond download, it
must specify the storage location, authority model, audit trail, and
validation tests.

## Accessibility and visual-language constraints

The MVP should preserve compatibility with the LRH Console visual-
language proposal and its Enhanced Swimlane Console direction.
Constraints:

- Do not use color alone to convey state.
- Every status must have visible text.
- Lanes need labels, counts, and explanations.
- Cards should use redundant cues such as text, icon/shape,
  border/accent, and count.
- Use semantic HTML landmarks where practical.
- Keep keyboard navigation in mind for lane navigation, card links,
  prompt text areas, copy buttons, and download controls.
- Use visual tokens rather than one-off colors.
- Preserve light/dark theme compatibility where the existing serve UI and
  console visual-language proposal provide tokens.
- Ensure prompt text remains selectable, scrollable, and readable without
  relying on hover-only controls.

## Implementation sequence

Recommended staged sequence:

1. **View-model and capability-gap contract.** Define serializable typed
   view models, deterministic lane classification, and capability-gap
   projection. Add unit tests before adding broad UI.
2. **Meta swimlane page.** Render registered projects into operational
   lanes using shared state. Include source state, validation status,
   counts, lane explanations, and capability gaps.
3. **Project operational dashboard.** Add project detail pages with
   validation, focus, design, workstream, work item, ready/not-ready leaf,
   and capability-gap summaries.
4. **Work item readiness page and prompt workbench.** Wire work item
   readiness details to the appropriate request renderer, with preview,
   copy, download, equivalent CLI, and repair prompt affordances.
5. **Design/workstream traceability pages.** Add drill-down pages that
   explain how designs, workstreams, work items, evidence, and capability
   gaps connect.

Each stage should be useful independently, preserve read-only behavior,
and keep implementation PRs small and reviewable.

## Validation / testing strategy

Future implementation PRs should include tests appropriate to their
stage:

- deterministic lane-classification tests;
- view-model serialization tests;
- capability-gap projection tests;
- action-affordance enable/disable tests;
- readiness badge tests;
- HTML escaping tests;
- route smoke tests;
- accessibility-oriented checks where practical;
- safety tests for localhost binding;
- tests that reject arbitrary file browsing; and
- tests that confirm no write endpoints exist in the MVP.

For documentation-only changes to this proposal, normal control-plane
validation such as `lrh validate`, `scripts/lint`, and other lightweight
repository checks are sufficient. Documentation PRs should not introduce
large unrelated fixes solely to address pre-existing failures.

## Risks and mitigations

| Risk | Mitigation |
| --- | --- |
| Serve duplicates planning semantics and drifts from CLI behavior. | Keep lane, readiness, prompt, and capability-gap interpretation in shared APIs with tests. |
| Users mistake prompt preview for autonomous execution. | Use explicit local-assist/human-gated language and avoid launch/send affordances. |
| Meta state appears to override project-local authority. | Label meta summaries as coordinating/informative and link to project-local source documents. |
| Planned gaps look like broken product state. | Model gap categories explicitly and reserve red/error styling for real errors and safety issues. |
| Prompt text exposes secrets or unexpected files. | Restrict source inputs to project-control artifacts and renderer-approved data; escape rendered HTML. |
| UI becomes inaccessible through color-only lanes. | Require labels, counts, explanations, semantic landmarks, keyboard navigation, and redundant cues. |
| Read-only MVP later accretes writes without review. | Require a separate write-boundary proposal, threat model, and tests for any future explicit-click writes. |

## Open questions

- Which existing planning/index API should own the first lane
  classifier, and should the classifier be exposed outside `serve` for
  snapshot or CLI diagnostics?
- What is the minimum project registry state needed to support cached or
  unavailable project cards without misleading users?
- How should adopted-but-not-implemented design counts handle partially
  implemented proposal sets with multiple sub-proposals?
- Should repair/readiness prompts be generated by a dedicated renderer or
  by parameterizing the existing work-item prompt renderer?
- What accessibility checks should be automated in the first server-route
  smoke tests?
- If a later write boundary is adopted, should it write prompt exports,
  execution records, run state, or all three through separate explicit
  actions?
