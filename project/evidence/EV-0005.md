# Release-smoke isolation audit closeout

The `WI-RELEASE-SMOKE-ISOLATION-AUDIT` work item is closed after review of the landed diagnostic-mode and optional strict-isolation behavior.

## Evidence

- `scripts/release-smoke <tag> --diagnose` is implemented and documented for printing pre-install temporary-venv isolation diagnostics.
- `scripts/release-smoke <tag> --strict-isolation` is implemented and documented for failing before wheel installation when LRH is visible in the temporary venv before the wheel under test is installed.
- Default `scripts/release-smoke <tag>` behavior remains warning-oriented when pre-install LRH visibility is detected, preserving local development usability while still surfacing isolation concerns.
- Smoke-venv commands run with `PYTHONPATH` cleared so the wrapper's source-checkout import path does not contaminate the installed-wheel validation or strict-isolation check.
- `scripts/release-smoke <tag> --diagnose --preserve` is documented as the investigation workflow for printing diagnostics and preserving the temporary smoke environment after exit.
- Focused unit tests cover command parsing, diagnostic collection/rendering helpers, pre-install visibility detection, default warning behavior, strict failure behavior, and cleanup/preservation behavior.

## Interpretation

The release-smoke policy is intentionally split:

- default mode validates installed-wheel behavior and warns about pre-install visibility rather than failing local development workflows;
- strict mode is available for CI, release audits, or maintainer checks that require a clean preinstall environment;
- diagnostic mode is available for explaining visibility sources without changing default pass/fail behavior.

Strict isolation is expected to fail in environments where LRH is already visible to the temporary venv before wheel installation. That failure is the intended strict-mode signal, not a prerequisite for closing this audit.
