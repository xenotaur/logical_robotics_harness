# LRH CI playbook dogfood and skill-readiness assessment

This record dogfoods the CI setup and debugging playbook at
`docs/project-setup/ci.md` against LRH itself. It is an assessment artifact only;
it does not implement a CI Agent Skill, add reusable workflow fragments, or modify
GitHub Actions workflows.

## 1. Project family classification

LRH is primarily a **Python package/tool** repository:

- package metadata and console script are defined in `pyproject.toml`;
- reusable package code lives under `src/lrh/`;
- tests live under `tests/`;
- maintainer command wrappers live under `scripts/`; and
- `.github/workflows/` contains GitHub Actions validation, coverage, smoke, and
  release lanes.

It also has a documentation/control-plane lane because `project/`, `docs/`, and
assist request templates are source artifacts that must remain human-readable and
validated by `lrh validate`.

## 2. Canonical LRH commands discovered

The playbook's LRH command categories fit the repository's existing wrapper
scripts. For LRH, the canonical local command set is:

| Category | Command | Dogfood finding |
| --- | --- | --- |
| Setup/bootstrap | `scripts/develop` | Setup-only command; should be run during environment bootstrap, not routine task-phase validation in Codex Cloud. |
| Tool/version report | `scripts/version tools` | Canonical first validation command; reports Python, Black, Ruff, PyYAML, and other relevant tool versions. |
| Workflow YAML check | `scripts/check-workflows` | Canonical local meta-CI check for `.github/workflows/*.yml`. |
| Format check | `scripts/format --check --diff` | Canonical non-mutating formatting gate. |
| Format fix | `scripts/format` | Canonical mutating format command for local repair. |
| Lint | `scripts/lint` | Canonical static-check command. |
| Tests | `scripts/test` | Canonical fast deterministic unit-test command. |
| Coverage | `scripts/coverage` or `scripts/coverage --html` | Available optional evidence command; CI coverage lane uses `scripts/coverage --html`. |

This confirms the playbook's example categories are not hypothetical for LRH;
they map directly to repository-owned scripts.

## 3. Setup versus validation guidance

The playbook's setup/validation split matches LRH's agent guidance:

- use `scripts/develop` during environment setup/bootstrap, where installing or
  reconciling dependencies is appropriate;
- during task-phase validation, start with `scripts/version tools`;
- if Black/Ruff versions are missing or mismatched, treat that as setup/cache
  drift before debugging formatter or linter output; and
- then run workflow checks, format checks, lint, and tests without routine
  dependency installation.

This boundary is actionable for LRH and should remain a required section in CI
assessment and implementation prompts.

## 4. Existing workflow coverage

LRH already has broad GitHub Actions coverage:

- `lint.yml` runs tool-version diagnostics, non-mutating format checks, and
  `scripts/lint` on pull requests and pushes to `main`.
- `tests.yml` runs tool-version diagnostics, `scripts/test`, and `lrh validate`
  on pull requests, pushes to `main`, and manual dispatch.
- `meta.yml` runs `scripts/check-workflows` on pull requests, pushes to `main`,
  and manual dispatch.
- `coverage.yml` runs `scripts/coverage --html` and uploads HTML coverage on
  pull requests, pushes to `main`, and manual dispatch.
- `installed-wheel-smoke.yml` runs strict installed-wheel smoke validation on
  pull requests, pushes to `main`, and manual dispatch.
- `smoke.yml` keeps heavier smoke validation separate on manual dispatch, tags,
  and a weekly schedule.
- `release-tag-ci.yml`, `release.yml`, and `testpypi-rehearsal.yml` cover tag
  validation, release artifact checks, PyPI publishing, and TestPyPI rehearsal
  paths outside the normal fast PR lane.

The workflow structure reflects the playbook's recommendation to keep fast PR
validation legible while separating heavier smoke, packaging, and release checks.

## 5. Playbook fit

What works well:

- The project-family discovery step correctly classifies LRH before proposing CI
  behavior.
- The command-category table maps cleanly to LRH's existing wrapper scripts.
- The setup-versus-validation section prevents agents from rerunning
  `scripts/develop` as ordinary task validation.
- The workflow-YAML guardrail is concrete and already wired into `meta.yml`.
- The evidence section matches LRH's existing preference for command output,
  version reports, and control-plane evidence records.

What is unclear or should be tightened after more dogfooding:

- The playbook says to replace LRH examples with repository-specific commands,
  but a future skill will need a more explicit evidence schema for recording
  absent categories such as coverage, smoke, or workflow checks.
- The playbook does not yet define a crisp abort/gate format for repositories
  whose setup cannot be reproduced locally.
- Request-template consumers may need examples for non-Python project families
  before automation can safely infer command categories.

## 6. Request-template readiness

The packaged CI request templates are directionally ready for further
dogfooding: `ci_assess_status` is read-only and explicitly applies the playbook,
while `ci_implement_workflow` carries the same command-semantics guidance into
implementation prompts.

They are not yet enough evidence for skill implementation. Before designing a CI
Agent Skill, the templates should be dogfooded against at least one non-LRH
repository family to verify that command discovery, abort handling, and reporting
stay repository-specific rather than Python/LRH-biased.

## 7. Skill-readiness recommendation

Do **not** proceed directly to CI Agent Skill implementation from this LRH-only
pass. The playbook is actionable for LRH, and the request templates appear ready
for broader dogfooding, but the skill design should wait until one more pass on a
different repository such as LCATS, Taurcode, Taurworks, or Velumin.

Recommended next step: run the read-only CI assessment prompt against one
non-LRH repository, record the resulting evidence, and then decide whether the
first skill artifact should be a design-only `SKILL.md` workflow, a reusable
assessment checklist, or no skill yet.

## 8. Follow-up recommendations

1. Dogfood `ci_assess_status` on one non-LRH repository and record whether the
   playbook handles absent or differently named setup/version/format/lint/test
   commands.
2. Add an explicit request-template reporting field for intentionally absent
   command categories before any skill automates the assessment shape.
3. Keep `scripts/develop` documented as setup-only in agent prompts and CI
   playbook guidance.
4. Keep workflow-template or reusable-fragment decisions deferred until at least
   two repository-family dogfood records exist.
5. If a CI Agent Skill is later designed, start with read-only assessment and
   evidence capture; do not make workflow-editing automation the first skill
   behavior.
